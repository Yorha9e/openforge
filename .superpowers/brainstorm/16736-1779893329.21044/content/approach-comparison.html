开关应该集成到 Profile 体系还是独立运行时系统？

你确认了开关控制的是「后端功能启用」——但实现方式有两种截然不同的路径。

方案 A：扩展 Profile YAML

在 minimal.yaml 等 Profile 文件中新增 feature_flags 字段：

profile: minimal
feature_flags:
  grafana_dashboards: false
  compliance_reports: false
  multi_region_dr: false
  k8s_container: false
  data_lifecycle: false
  notifier: stdout    # 已有字段，不变

✅ 优点

  • 复用现有 Profile 加载机制，零新增基础设施
  • 启动时一次性决策，行为可预测
  • 变更需重启，天然防止误操作

⚠️ 缺点

  • 切换开关需要修改 YAML + 重启服务
  • 无法在 Admin 面板中实时切换（需重启生效）
  • 多人部署时配置文件同步问题
方案 B：运行时 Feature Flags (DB)

新建 feature_flags 表，Admin 面板直接读写：

CREATE TABLE feature_flags (
  name  TEXT PRIMARY KEY,
  enabled BOOLEAN DEFAULT false,
  profile_min TEXT, -- 最低 Profile
  updated_at TIMESTAMPTZ
);

✅ 优点

  • Admin 面板开关 → 实时生效，无需重启
  • 多实例共享 DB，天然一致
  • 可做审计日志（谁在什么时候改了什么）

⚠️ 缺点

  • 新增 DB 表和 API 端点
  • 部分功能（如 K8s Container Runtime）切换需重建连接池
  • 运行时切换有状态一致性问题
A

扩展 Profile YAML

个人模式用户直接设置 minimal.yaml，企业部署用 enterprise.yaml，在配置文件中声明开关

B

运行时 Feature Flags

新建 feature_flags 表 + API，Admin 面板实时切换，不用重启

C

混合方案

Profile 设定默认值 + DB 覆盖。minimal 默认全关，enterprise 默认全开，Admin 面板可覆盖
